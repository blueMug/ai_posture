# 出片姿势小程序需求总结

本文档记录当前小程序已经沉淀出的产品、资源和拍照页配置要求。该文档仅用于项目说明，不参与小程序打包。

## 1. 产品定位

`出片姿势` 是一款按场景推荐拍照姿势的小程序。用户先根据场景选择姿势模板，再进入姿势相机，照着轮廓图或半透明真人图拍照。

核心目标：

- 按自拍、穿搭、旅行、街拍、咖啡馆、背影等场景推荐姿势。
- 每个姿势提供真人图、缩略图、轮廓图和动作说明。
- 相机页叠加姿势引导，帮助用户对齐动作。
- 不做人像识别、姿势识别或实时纠偏。
- 收藏、使用记录和相机偏好保存在本地。

## 2. 页面结构

底部 Tab 包含 3 个页面：

- `姿势推荐`：首页，负责推荐、搜索、场景导演和快捷开拍。
- `拍照场景`：姿势分类页，负责按分类展示全部姿势。
- `个人中心`：负责收藏、相机默认设置、隐私和反馈入口。

非 Tab 页面：

- `隐私与权限说明`
- `意见反馈`
- `姿势详情`
- `拍照页`
- `拍后预览页`

整体视觉要求：

- 深色背景。
- 主强调色为黄色 `#ffe95c`。
- 小程序窗口使用自定义导航。

## 3. 首页要求

首页不是普通瀑布流，而是“场景推荐入口”。

### 3.1 顶部品牌

顶部展示文字品牌：

- 品牌名：`出片姿势`
- 标语：`按场景推荐拍照姿势，照着模板轻松出片`

当前首页代码没有实际展示 logo 图片。`assets/brand` 中存在品牌资源，但首页实际使用文字品牌。

### 3.2 搜索

首页顶部需要搜索框。

占位文案：

`搜索拍照姿势、姿势相机、旅行、街拍、穿搭`

搜索逻辑：

- 输入关键词后，按姿势名称、说明、分类和搜索关键词匹配。
- 没有结果时提示：
  - 标题：`没有匹配姿势`
  - 说明：`试试“半身”“自拍”“全身”“背影”。`

### 3.3 场景拍照导演

首页首屏核心模块是 `场景拍照导演`。

模块文案：

- Kicker：`场景拍照导演`
- 标题：`先选你在哪，再决定怎么拍`

场景 Tab 当前有 5 个：

- `咖啡店`
- `旅行景点`
- `街边路口`
- `穿搭全身`
- `自拍近景`

每个场景 Tab 使用一张姿势封面图。封面图优先使用首页本地展示图，否则走 CDN。

### 3.4 场景拍照方案

每个场景下最多展示 3 条拍摄方案。

每条方案包含：

- 缩略图
- 标签
- 标题
- 推荐原因
- `照着拍`按钮

方案内容包含：

- `reason`：为什么适合该场景。
- `composition`：构图建议。
- `camera`：机位建议。
- `poseCue`：动作提示。
- `avoid`：避坑提示。

排序与筛选规则：

- 优先从场景候选姿势中取姿势。
- 优先使用首页本地资源池内的姿势，保证首屏稳定性。
- 只使用同时具备真人图和详情图的姿势。
- 按内容丰富度排序，最多取 3 个。
- 如果命中手写方案，则使用手写标题、标签、动作、构图和避坑文案。
- 场景 Tab 的封面姿势不进入方案列表，避免重复。

### 3.5 今日推荐

首页需要 `今日推荐` 模块，每天轮换一个主题。

轮换主题顺序：

1. `今天拍穿搭`
2. `今天试试不露脸`
3. `今天拍咖啡馆`
4. `今天自拍不尴尬`
5. `今天出门街拍`
6. `今天旅行打卡`
7. `今天坐着也好拍`

展示规则：

- 根据当天本地日期计算轮换主题。
- 每个主题展示 4 个姿势。
- 优先展示首页本地资源池中的姿势。
- 每张卡片包含图片、姿势名和 `照着拍` 按钮。

### 3.6 马上开拍

首页需要 `马上开拍` 模块。

固定 4 个高频入口：

- `拍穿搭`：进入穿搭全身类，封面使用该类中代表性强、适合全身比例展示的姿势。
- `旅行打卡`：进入旅行打卡类，封面使用该类中场景感强、适合景点直接照拍的姿势。
- `不露脸`：进入背影/不露脸类，封面使用背影、回眸或转身类姿势。
- `自拍`：进入自拍类，封面使用近景自然、表情明确的自拍姿势。

点击后进入对应分类。

### 3.7 更多拍照灵感

首页需要 `更多拍照灵感` 模块。

默认展示 4 类推荐：

- `今天拍穿搭`
- `头像半身照`
- `不露脸背影`
- `自拍不尴尬`

每类最多展示 4 个姿势，并且会跨分类去重。

## 4. 图片资源要求

### 4.1 图片类型

项目中同一姿势通常涉及以下图片字段：

- `guideImage`：透明 PNG 轮廓图，用于相机轮廓引导。
- `thumbnailImage`：缩略图，用于列表和卡片。
- `modelImage`：真人参考图，用于半透明照片模式和分类页展示。
- `detailImage`：详情图，用于详情页和部分卡片封面。

### 4.2 本地优先资源策略

图片资源整体遵循“本地优先，本地没有再走 CDN”的规则。

通用要求：

- `guideImage`、`thumbnailImage`、`modelImage`、`detailImage` 都应先判断是否存在可用本地资源。
- 可用本地资源是指未被 `packOptions.ignore` 排除、会实际进入小程序包的文件。
- 本地存在且实际打包的图片和轮廓图直接使用 `/static/...` 本地路径。
- 本地缺失、包体不适合放入本地、或资源配置中明确声明为远程的图片，才使用 CDN 地址。
- 资源兜底应尽量收敛到具体文件，不应因为某个姿势缺少一张图就强制整组图片都走 CDN。
- 同一姿势中，可用本地轮廓图可以继续走本地，本地缺失或未打包的真人图、缩略图、轮廓图单独走 CDN。
- 姿势详情页大图如果本地路径加载失败，需要自动切换到对应 CDN 地址重试。

首页首屏和推荐模块还需要使用本地轻量图，减少加载不稳定。

本地首页资源目录：

- `static/recommend_thumbs/`
- `static/home_guides/`

本地首页图片映射规则：

- 轮廓图从 `static/pose_pairs`、`static/pose_guides` 或 `static/recommend_guides` 映射到 `static/home_guides`。
- 缩略图从 `static/pose_pairs`、`static/pose_thumbs` 或 `static/recommend_thumbs` 映射到 `static/recommend_thumbs`。
- `_demo.jpg` 会映射为 `_thumb.jpg`。

首页本地资源池要求：

- 首页本地缩略图可以覆盖较多推荐姿势，因为 `static/recommend_thumbs/` 体积较小，优先保障首页图片稳定显示。
- 首页本地轮廓必须严格控制数量，避免主包超过 2MB。
- 本地轮廓只保留用户最容易直接点进拍照的高曝光入口：默认场景首屏推荐和当前高曝光今日推荐。
- 当前实现中，本地轮廓白名单由 `utils/assets.js` 的 `HOME_LOCAL_GUIDE_FOLDERS` 控制，目前只保留 7 张高曝光轮廓。
- 当前 7 张本地轮廓为：`custom76`、`custom78`、`custom80`、`custom84`、`custom88`、`custom103`、`custom105`。如果首页默认场景或今日推荐策略调整，需要同步评估并更新这个集合。
- 不在 `HOME_LOCAL_GUIDE_FOLDERS` 内的首页姿势，即使缩略图在本地，点击拍照后的轮廓也应走 CDN。
- 放入本地轮廓池的姿势必须同时具备轻量缩略图和轮廓图。
- 首页本地资源优先保障轻量缩略图，避免首屏图片空白。
- 如果某张轮廓放入主包后会导致超过 2MB，优先把该轮廓移出本地白名单，继续走 CDN。
- 首页出现过的姿势，即使用户没有点进详情页，也需要在后台预加载详情大图；远程图应依赖缓存流程提升进入详情页和拍照页速度。

### 4.3 CDN 资源

默认远程资源走 jsDelivr：

`https://cdn.jsdelivr.net/gh/blueMug/posture_assets@main`

CDN 是本地缺失资源的兜底来源，不再作为所有非首页图片的默认来源。

如有特殊图片需要强制远程加载，应只在资源配置中按文件声明，避免按姿势整组扩大影响范围，也避免在需求文档里写死具体姿势编号。

### 4.4 不同页面的图片优先级

以下优先级指字段选择顺序。每个字段在解析最终 URL 时仍遵循本地优先、CDN 兜底。

首页推荐卡片：

1. 首页本地 `recommend_thumbs`
2. `detailImage`
3. `modelImage`
4. `thumbnailImage`
5. `guideImage`

姿势分类页：

1. `modelImage`
2. `detailImage`
3. `thumbnailImage`

姿势详情页：

1. `detailImage`
2. `thumbnailImage`
3. `guideImage`

拍照页：

- 轮廓模式使用 `guideImage`。
- 半透明照片模式使用 `modelImage`。

分享封面：

1. `thumbnailImage`
2. `detailImage`
3. `guideImage`

### 4.5 图片和轮廓制作要求

每个可拍姿势建议同时准备真人参考图、详情图、缩略图和透明轮廓图。

真人参考图要求：

- 人物姿势必须清晰，手、脚、头部朝向和身体重心要明确。
- 构图应尽量完整，尤其是全身、坐姿、蹲姿、背影类姿势不要裁掉脚、手臂或关键道具。
- 背景可以有场景氛围，但不能抢过人物动作；姿势库图需要让用户一眼看出“照着怎么摆”。
- 自拍和半身图要保留脸、肩颈、手部动作；不要让道具或手完全遮挡脸部关键表情。
- 全身图建议保留头顶和脚底空间，便于相机页轮廓对齐。
- 同一姿势的真人图、详情图、缩略图和轮廓图要使用同一动作，避免用户在详情页和拍照页看到的姿势不一致。

轮廓图要求：

- 使用透明 PNG。
- 只保留对齐所需的人体外轮廓和关键道具轮廓，背景必须透明。
- 轮廓要覆盖头部、肩线、手臂、手部、躯干、腿部、脚部等关键姿态信息。
- 道具只有在动作依赖它时才保留，例如咖啡杯、相机、书、伞、帽子等。
- 线条应足够清楚，在相机预览上叠加后仍可辨认；轮廓不应过细、过淡或有明显毛边。
- 轮廓图需要和真人参考图保持相同姿势比例和主体位置，不能随意拉伸。
- 轮廓图应尽量裁到主体边界附近，但保留少量透明边距，避免贴边影响相机页 `aspectFit` 布局。
- 全身姿势必须保留脚部轮廓；半身和自拍姿势必须保留手部、肩颈和脸部轮廓。

缩略图要求：

- 首页轻量缩略图用于快速加载，尺寸和体积应明显小于详情图。
- 缩略图要优先呈现人物动作，不应只截取环境或局部。
- 列表图可以使用真人参考图裁切，但需要保证动作识别度。

详情图要求：

- 详情图用于放大查看姿势，应比缩略图更清晰。
- 需要完整呈现动作要点，并和拍照页轮廓保持一致。

命名和目录要求：

- 同一姿势的缩略图、轮廓图、真人图和详情图命名要能互相对应。
- 首页本地轻量图放入 `static/recommend_thumbs/`，首页拍照入口需要本地加载的轮廓放入 `static/home_guides/`。
- 姿势原始展示图和轮廓图按现有 `static/pose_*`、`assets/pose_pairs` 等目录组织。
- 新增资源时，应同步更新姿势元数据中的 `guideImage`、`thumbnailImage`、`modelImage` 和 `detailImage` 字段。

### 4.6 图片和轮廓的后台加载方式

后台加载目标：

- 首页首屏优先用本地轻量资源，减少白屏和图片加载失败。
- 预加载策略必须服务首次使用链路：新人打开小程序后，应能尽快看到首页、选择姿势并进入拍照。
- 拍照必需资源优先级高于详情大图、全量分类图和广告以外的非关键资源。
- 远程高清图在后台逐步缓存，不阻塞用户进入页面。
- 收藏姿势的关键图片要固定缓存，提升下次打开和离线场景的稳定性。

启动预加载：

- 小程序启动时调用图片预加载流程。
- 启动阶段不应抢占首屏和首次拍照资源，不应在启动后立即下载大量详情大图。
- 全量姿势分类图只能在较长延迟后顺序预加载，用于空闲补缓存。
- 预加载按顺序逐张缓存，避免同时发起过多下载请求。
- 预加载只负责远程图片，本地图片直接使用本地路径。
- 首页轮廓预加载必须先经过 `homeLocalAssetUrl()` 解析：解析为 `/static/home_guides/...` 的本地轮廓不进入下载队列，解析为 CDN 的轮廓才进入远程预加载。
- 不要为了 CDN 兜底而给本地白名单轮廓额外预下载同一张远程图；本地加载失败时由拍照页失败兜底处理。

页面按需缓存：

- 首页在构建推荐、今日推荐和场景导演后，先缓存即将展示的缩略图。
- 首页可见姿势的拍照引导图优先后台缓存，保证用户点击“拍照”后轮廓或半透明参考能更快出现。
- 首页可见姿势的详情大图延后后台缓存，避免和首屏图片、拍照引导图抢资源。
- 姿势详情页打开时，先缓存详情展示图，同时预取拍照页需要的轮廓图。
- 拍照页进入时，只等待当前实际要显示的引导图；缩略图和真人图等辅助资源后台补缓存，不能阻塞相机取景和引导层渲染。
- 场景导演的拍摄方案图会在后台缓存，缓存完成后替换为本地 saved file 路径。

缓存实现规则：

- 只缓存 HTTP/HTTPS 远程图片；本地路径直接返回。
- `queueImagePreload()` 应过滤非远程 URL，避免把 `/static/home_guides/...`、`/static/recommend_thumbs/...` 等本地路径送入下载流程。
- 后台预加载必须进入统一队列顺序执行，避免启动、首页、分类页各自开队列造成并发抢资源。
- 用户主动进入详情页或拍照页时，当前页面必需图片可以直接加载，不应被后台预加载队列阻塞。
- 下载成功后使用微信 `saveFile` 保存到本地文件系统。
- 缓存索引存入本地 storage，记录原始 URL、本地文件路径、文件大小和最后访问时间。
- 同一 URL 同时被多处请求时，应复用同一个 pending 下载任务，避免重复下载。
- 读取缓存时先校验本地 saved file 是否仍存在；不存在则删除索引并重新下载。
- 普通缓存总量上限约为 `8MB`。
- 超过上限时按最久未访问优先清理，但不能清理被 pin 的收藏图片。

收藏图片缓存：

- 用户收藏姿势时，需要缓存并 pin 住该姿势的 `guideImage`、`thumbnailImage`、`modelImage`、`detailImage`。
- pin 住的图片不参与普通缓存淘汰。
- 已收藏姿势再次展示、进入详情页或进入拍照页时，必须优先读取本地 saved file 缓存；本地缓存存在时不能继续依赖 CDN。
- 对于历史已收藏但还没有本地缓存的姿势，进入个人中心、详情页或拍照页时需要补齐缓存并 pin 住。
- 收藏姿势的拍照引导图必须优先使用本地缓存的轮廓图或真人参考图，确保收藏后的核心拍照链路不依赖 CDN。
- 用户取消收藏时，需要解除这些图片的 pin；后续可被普通缓存清理。

失败兜底：

- 本地包内图片加载失败时，应优先重试对应 CDN 地址，避免因本地文件缺失或路径异常导致页面不可用。
- 拍照页如果 `/static/home_guides/...` 本地轮廓加载失败，应使用同名 CDN 轮廓重试，而不是直接停留在失败状态。
- 图片下载失败时返回原远程 URL，页面仍尝试直接加载远程图。
- 缓存写入失败不应阻塞页面展示。
- 姿势详情页大图加载期间必须展示明确的加载中状态，例如半透明遮罩、转圈图标和“图片加载中”文案；图片 `load` 或 `error` 后需要关闭加载态。
- 拍照页轮廓加载失败时需要提示用户“轮廓图加载失败”。
- 列表图片加载失败时需要显示失败提示，并支持重试或重新拼接重试参数。

## 5. 姿势分类和排序

姿势基础数据由资源清单和姿势元数据共同决定。具体姿势编号会随着素材迭代变化，不应在需求文档中固定。

当前分类共 10 类：

- `今天拍穿搭`
- `头像半身照`
- `出门街拍`
- `旅行打卡`
- `不露脸背影`
- `咖啡馆/道具`
- `自拍不尴尬`
- `坐着也好拍`
- `室内窗边`
- `展馆/城市建筑`

分类内容由姿势配置指定，需求文档只描述分类语义，不固定具体姿势编号。

分类内排序不是简单按编号，而是按内容丰富度排序：

- `description` 权重最高。
- `tip` 次之。
- `name` 次之。
- 姿势的新鲜度或资源排序也可以作为辅助加分项。
- 如果分数相同，可以按配置中的稳定兜底排序处理。

首页推荐分类还需要去重，前面已经展示过的姿势，后续分类不再重复展示。

## 6. 拍照页要求

拍照页是核心功能，不是普通相机。

### 6.1 权限和隐私

进入相机前必须确认隐私说明。

隐私说明覆盖：

- 相机：用于拍摄带姿势参考的照片。
- 保存相册：仅在用户点击保存时保存照片。
- 本地收藏：保存用户收藏的姿势。
- 本地使用记录：记录浏览和拍照行为。
- 意见反馈：用户主动提交后由腾讯问卷收集。
- 除意见反馈外，收藏和使用记录仅保存在本机。

用户不同意隐私说明时，不进入相机。

### 6.2 相机基础配置

相机使用微信原生 `<camera>`。

配置：

- `resolution="high"`
- `flash="off"`
- 默认缩放 `1x`
- 默认最大缩放兜底为 `10x`
- 初始化后读取设备实际 `maxZoom`
- 默认画幅比例为 `3:4`
- 设置面板支持在 `3:4` 和 `9:16` 两种画幅间切换，并写入本地存储。
- `9:16` 只改变相机预览画幅，不改变轮廓图的参考区域和默认大小。
- 如果设备高度不足以完整展示 `9:16`，相机预览可以等比缩窄并居中，页面背景保持黑色，保留上下或左右黑边，不能拉伸画面。

摄像头默认选择：

- 普通姿势默认后置。
- 如果姿势名称、提示、说明或分类中包含 `自拍`，默认前置。

切换前后摄像头后，需要把缩放重置为 `1x`。

### 6.3 引导模式

相机支持两种引导模式：

- `轮廓`
- `半透明照片`

默认使用轮廓。

用户可在个人中心设置默认引导模式，也可在拍照页临时切换。

如果当前姿势没有 `modelImage`，半透明照片模式自动退回轮廓模式。

显示透明度：

- 轮廓模式透明度约 `0.92`
- 半透明照片模式透明度约 `0.42`

### 6.4 引导图交互

姿势引导图需要支持：

- 单指拖动。
- 双指缩放。
- 设置面板中使用 `-` 和 `+` 微调大小。
- 点击百分比重置为 `100%`。
- 点击 `旋转` 每次顺时针旋转 `90°`，按 `0° -> 90° -> 180° -> 270° -> 0°` 循环。

限制：

- 缩放范围：`35%` 到 `220%`
- 每次按钮缩放步进：`10%`
- 横向拖动范围：约 `±120px`
- 纵向拖动范围：约 `±160px`
- 旋转、缩放和拖动状态需要同时影响拍照页展示、拍后带轮廓确认和保存合成图。
- 拍照页不提供轮廓镜像功能；此前真机上的 `cover-image` CSS 负向缩放和隐藏 canvas 临时图方案表现不稳定，容易出现翻转方向错误或轮廓消失。
- 有轮廓地址但图片尚未加载完成时，在相机画面中央显示动态加载图标和 `轮廓加载中` 文案；加载成功或失败后隐藏。

进入相机后需要展示一次提示：

- 标题：`姿势可以拖动对齐`
- 说明：`点左上设置，可调整轮廓大小和倒计时`

提示只在当前会话显示一次，约 3.2 秒后自动消失。

### 6.5 引导图布局

拍照页底部面板高度按 `330rpx` 计算。

引导布局规则：

- 相机预览在底部控制区上方的可用空间内垂直居中，并可按设计向下微调。
- 页面背景保持黑色，相机预览没有占满屏幕时保留黑边，接近普通相机观感。
- `3:4` 下相机预览宽高比为 `3:4`。
- `9:16` 下相机预览宽高比为 `9:16`。
- 无论相机画幅是 `3:4` 还是 `9:16`，轮廓参考区域都固定按 `3:4` 计算，避免切到 `9:16` 后轮廓被拉长、变窄或整体缩放。
- 舞台顶部约为 3:4 参考区域高度的 `8%`。
- 舞台底部预留约为 3:4 参考区域高度的 `4%`。
- 引导区域左边距约为窗口宽度的 `7%`。
- 引导区域宽度约为窗口宽度的 `86%`。
- 引导图在舞台内顶部约为舞台高度的 `8%`。
- 引导图高度约为舞台高度的 `88%`。
- 根据图片宽高使用 `aspectFit` 计算实际引导框。

轮廓比例防错规则：

- 这里的 `3:4` 只指相机预览和轮廓参考区域，不代表每一张轮廓图片自身都必须是 `3:4` 画布。
- 拍照页轮廓图片必须保持资源本身的宽高比。窄站姿、背影、坐姿、趴姿等裁切轮廓，不能为了凑 `3:4` 画布而补宽或改用 `static/pose_pairs` 的整框图。
- 首帧布局和图片加载后的布局必须使用同一张 active `guideImage` 的同一组尺寸。首帧来自 `utils/guideImageSizes.js`，加载后来自 `wx.getImageInfo`；两者宽高比不一致时，用户会看到刚进拍照页一版比例、加载完成后又跳到另一版比例。
- 如果某个姿势的拍照引导应该使用裁切轮廓，必须在姿势元数据里显式设置 `guideImage: '/static/pose_guides/..._contour.png'`，不能依赖默认 `static/pose_pairs` 路径。
- `static/pose_pairs` 可以是为了列表、组合或兜底准备的整框资源；它不一定适合作为拍照页 active 引导图。
- 只允许在确认“画布补边不会改变拍照页 active 轮廓视觉比例”时才给 PNG 补透明边。禁止把“外框变成 `3:4`”当作比例修复完成。
- CDN 资源和本地尺寸表存在时间差时，风险更高：如果本地 PNG 已改、`guideImageSizes` 也已按本地新图生成，但实际加载的 CDN 仍是旧比例，就会再次出现首帧和加载后不一致。

已发生过的错误和修复结论：

- 错误 1：只修了 `wx.getImageInfo` 回来后的布局，没有给首帧提供可靠尺寸，导致未加载完成时可能按容器铺满。修复：拍照页先用 `getGuideImageSize()` 查 active `guideImage` 尺寸，再用真实图片尺寸回填。
- 错误 2：把 `9:16` 下轮廓参考区域固定 `3:4` 误解成所有轮廓资源都要改成 `3:4`，错误补宽了部分裁切轮廓。修复：恢复裁切轮廓比例，只让参考区域固定 `3:4`。
- 错误 3：只跑本地资源比例检查，没比较 active `guideImage`、尺寸表和当前基线/CDN 可能加载的旧图，导致 `湖畔草帽回眸` 这类姿势首帧按本地整框图，加载后按裁切图。修复：显式指定 `guideImage` 为裁切轮廓，并重新生成尺寸表。

轮廓比例问题排查流程：

1. 先定位用户反馈的姿势名称对应的 `pose.id` 和 active `guideImage`，不要直接按资源目录猜测。
2. 检查拍照页首帧 `pose-guide-box` 宽高比是否来自 `utils/guideImageSizes.js`，再检查加载后的 `wx.getImageInfo` 宽高比是否一致。
3. 同时比较三份数据：active `guideImage` 本地 PNG 头、`utils/guideImageSizes.js` 记录、当前基线或 CDN 仍可能加载到的同路径 PNG。
4. 对裁切轮廓再看透明像素主体边界，不能只看 PNG 画布宽高。主体比例正确但画布被补宽，也会造成视觉比例错误。
5. 如需修复，优先修姿势元数据的 `guideImage` 指向和尺寸表；只有确认 active 资源本身确实错了，才修改 PNG。
6. 修复后必须运行 `node scripts/check-guide-ratios.js`，再运行 `node --check pages/camera/index.js`、`node --check utils/guideImageSizes.js` 和 `git diff --check`。

- `轮廓大小`
- `引导`
- `拍后带轮廓`
- `倒计时`
- `旋转`

倒计时选项：

### 6.7 拍摄提示

拍照页不再展示长段拍摄提示卡，也不在设置面板保留 `拍摄提示` 开关。

原因：

- 拍照页运行在微信原生 `<camera>` 层上，复杂 `cover-view` 文本和普通 `view` 叠层在真机上容易出现裁剪、省略号、错位或白块。
- 用户进入拍照页时核心任务是对齐轮廓并拍照，长提示应放在首页场景方案和姿势详情页。

- 标题：`姿势可以拖动对齐`
- 说明：`点左上设置，可调整轮廓大小和倒计时`

### 6.8 拍照和保存流程

点击快门后：

- 如果倒计时关闭，立即拍照。
- 如果倒计时开启，倒计时结束后拍照。

拍照质量使用 `high`。

未开启 `拍后带轮廓` 时：

- 拍完直接保存到系统相册。
- 保存成功后停留在相机页继续拍。
- 底部显示本次最近照片缩略图和本次拍摄数量。
- 点击最近照片直接打开系统全图预览，从本次第一张照片开始，支持左右滑动查看本次拍摄照片。
- 系统全图预览使用微信原生 `wx.previewImage`，不在预览层上叠加自定义分享按钮。

开启 `拍后带轮廓` 时：

- 拍完进入预览页。
- 预览页可以开关轮廓显示。
- 用户可以放弃、分享或保存。
- 保存时如果轮廓可见，会把轮廓合成到输出图。
- 预览页轮廓必须复用拍照页当前的缩放、拖动和旋转结果。

合成限制：

- 最大画布边长为 `1600`。
- 合成失败时回退保存原照片。

## 11. 文字展示要求

整体文案目标：

- 文案要帮助用户更快完成“选姿势、看动作、照着拍、保存照片”。
- 首页和详情页可以承担解释与推荐，拍照页只承担即时动作提示。
- 文案应直接、具体、可执行，少用抽象形容词。
- 同一含义的按钮文案要保持一致，避免同一操作在不同页面出现多个叫法。

### 11.1 首页文案

首页文案可以相对完整，但仍要保持短句。

要求：

- 品牌标语说明小程序用途，不展开过多功能细节。
- 搜索占位词应覆盖高频搜索意图，例如姿势、旅行、街拍、穿搭、自拍等。
- 场景导演标题要表达“先选场景，再决定怎么拍”的决策路径。
- 场景 Tab 文案使用真实生活场景，如咖啡店、旅行景点、街边路口、穿搭全身、自拍近景。
- 今日推荐标题要有当天行动感，例如“今天拍穿搭”“今天旅行打卡”。
- 首页卡片按钮优先使用 `照着拍`，强调马上进入相机对齐模板。
- 分类入口文案要说明结果，例如“显高显比例”“景点直接照着站”“背影回眸不尴尬”。

场景方案文案要求：

- `title`：短标题，说明动作或场景，不超过一行。
- `badge`：短标签，概括风格或拍摄收益。
- `reason`：解释为什么适合当前场景，可以稍完整。
- `composition`：说明画面怎么框，优先写“半身入镜”“全身入镜”“保留脚部/手部/道具”。
- `camera`：说明手机高度和方向，优先写“平视”“略低”“略高”“竖拍”。
- `poseCue`：说明用户怎么摆，必须是可执行动作。
- `avoid`：说明一个最容易犯的错误，语气直接。

### 11.2 姿势分类页文案

分类页以快速浏览为主，文字要少。

要求：

- 页面标题使用 `姿势分类`。
- 搜索占位词使用短关键词组合，不写长句。
- 分类导航只展示分类名和数量。
- 图片卡片只展示姿势名和短操作按钮。
- 拍照按钮可使用 `拍`，比 `照着拍` 更节省卡片空间。
- 图片加载失败时使用明确可操作文案，例如 `图片加载失败，点击重试`。
- 空状态文案要给出可搜索示例，例如“试试‘半身’‘自拍’‘景点’‘咖啡’”。

### 11.3 姿势详情页文案

详情页可以展示更完整的动作说明，但需要分层。

要求：

- 首屏优先展示姿势名、大图和进入拍照的入口。
- 图片旁的提示使用 `点图拍照` 这类短文案。
- `适合` 文案说明场景或拍摄收益，不放太长动作段落。
- `动作要点` 最多展示 3 条，每条应是一个动作，不要混合多个层级。
- `详细说明` 默认可收起，避免详情页过长。
- 分享按钮使用 `分享同款`，拍照按钮使用 `照着拍`。

姿势元数据文案要求：

- `name`：短、具体，可描述动作或画面，例如“侧身撩发”“背影行走”“台阶坐姿”。
- `tip`：说明适合场景或拍摄收益，适合列表和详情摘要。
- `description`：可以较完整，但要能被拆成动作要点；尽量包含动作、视线、手部、脚部、构图或注意事项。
- 搜索关键词应覆盖用户可能输入的场景、动作、道具和风格词。

### 11.4 拍照页文案

拍照页文案必须简单，用户打开相机时不应读长段文字。

原则：

- 只展示当前拍照必要信息。
- 优先使用短词、短句和状态词。
- 不展示长段解释，不展示完整推荐理由。
- 一次只提示 1 到 2 个动作重点。
- 文案要贴近操作，例如“动作”“构图”“轮廓大小”“倒计时 3s”。

拍照页固定文案要求：

- 设置面板标题：`拍摄设置`
- 缩放设置：`轮廓大小`
- 引导开关：`引导 轮廓`、`引导 半透明照片`、`引导 关`
- 拍后确认：`拍后带轮廓 开`、`拍后带轮廓 关`
- 倒计时：`倒计时 关`、`倒计时 3s`、`倒计时 5s`、`倒计时 10s`
- 拍摄提示：`拍摄提示 开`、`拍摄提示 关`
- 提示标签：`动作`、`构图`
- 选姿势入口：`选姿势`

拍摄提示内容要求：

- `动作` 文案只写用户身体怎么摆，例如“一手端杯，另一手撩发”。
- `构图` 文案只写画面怎么取，例如“半身入镜，保留杯子和手部动作”。
- 每条提示建议控制在一行内，尽量不超过 20 个汉字。
- 如果来自场景导演，只取 `poseCue` 和 `composition`，不展示 `reason`、`camera`、`avoid` 的长说明。
- 如果来自普通姿势说明，需要抽取最关键动作，避免把完整 `description` 原样塞进相机页。
- 拍照页不展示营销式、解释式文案，例如“为什么推荐”“适合什么氛围”等。

拍照页一次性引导提示：

- 标题：`姿势可以拖动对齐`
- 说明：`点左上设置，可调整轮廓大小和倒计时`
- 该提示只在当前会话显示一次，并自动消失。

拍照页错误和状态文案：

- 轮廓加载失败：`轮廓图加载失败`
- 相机不可用：展示系统错误或 `相机不可用`
- 拍照失败：`拍照失败`
- 保存成功：`已保存，继续拍`
- 保存失败：`保存失败`
- 相册授权失败：`请授权保存到相册`
- 本次没有照片：`还没有本次照片`

### 11.5 拍后预览页文案

拍后预览页要围绕确认、保存、分享和重拍。

要求：

- 普通预览标题使用 `照片预览`。
- 带轮廓确认时标题使用 `确认轮廓`。
- 轮廓开关使用 `关轮廓` 和 `开轮廓`。
- 操作按钮使用 `放弃`、`分享`、`保存`。
- 普通预览底部使用 `分享照片`、`保存到相簿`。
- 合成失败提示使用 `合成轮廓失败`，并允许回退保存原照片。

### 11.6 个人中心、隐私和反馈文案

个人中心文案要解释设置影响，但不写技术细节。

要求：

- 收藏区说明用户已收藏数量，并说明可直接查看或拍照。
- 相机引导样式说明“默认使用轮廓，半透明照片适合需要对照真人图时开启”。
- 拍后确认轮廓说明“拍完先看轮廓对齐效果”。
- 默认显示拍摄提示说明“进入拍照页会默认显示动作和构图提示，拍照页内仍可临时关闭”。
- 隐私说明需要明确相机、保存相册、本地收藏、本地使用记录和意见反馈的用途。
- 意见反馈说明需要明确“复制腾讯问卷链接，填写后内容会同步给创作者”。

隐私弹窗文案要求：

- 使用功能前先说明权限用途。
- 确认按钮使用 `同意继续`。
- 取消按钮使用 `暂不使用`。
- 文案要明确除用户主动提交的反馈外，收藏和使用记录仅保存在本机。

反馈页文案要求：

- 明确当前环境需要复制链接后打开腾讯问卷。
- 复制成功后提示用户到微信或浏览器粘贴打开。
- 如果链接未配置，需要明确提示开发者配置 HTTPS 腾讯问卷链接。

### 11.7 Toast、空状态和错误文案

Toast 文案要求：

- 成功类短而明确，例如 `已保存`、`已分享`、`已收藏，缓存中`。
- 失败类说明具体失败点，例如 `图片加载失败`、`预览失败`、`复制失败，请稍后重试`。
- 权限类说明下一步，例如 `请授权保存到相册`。

空状态文案要求：

- 标题说明当前状态。
- 描述给出下一步建议。
- 搜索空状态必须给出可尝试关键词。

错误文案要求：

- 能重试的错误要明确“点击重试”。
- 不能立即修复的错误要说明原因或下一步，例如“请检查图片域名”“去设置”。

按钮文案要求：

- 首页进入相机：`照着拍`
- 列表紧凑拍照入口：`拍`
- 详情页拍照入口：`照着拍`
- 返回：`返回`
- 清除搜索：`清除`
- 查看更多：`更多` 或 `看全部`
- 收藏：`收藏`、`已收藏`
- 删除收藏：`取消`
